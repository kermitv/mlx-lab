What Problems Were Run, And What Happened

How To Use This Page

Runs Are Not Identical

The benchmark program evolved across runs. Early runs were simpler and more exploratory; later runs were more structured and closer to a routing benchmark.

Task Names Matter

Each run used a small set of benchmark problems. Those problem definitions are what determine whether a score or model recommendation means anything.

Run 4 Is The Most Structured

Run 4 has the cleanest prompt set, normalized data layer, saved raw outputs, and chart-backed interpretation. Runs 1 to 3 are lighter and should be read accordingly.

Run 1

Run 1 was the baseline viability run. It asked whether a local MLX-served model on oc-mac-m5 was already practically useful before any deeper routing or escalation logic was introduced.

Models In Scope

Qwen2.5-7B-Instruct-4bit, Qwen2.5-32B-Instruct-4bit, and Qwen2.5-Coder-32B-Instruct-4bit.

Problems Run

  • Throughput: streamed local generation benchmark focused on TTFT and tokens per second.
  • Coding: generate a real scripts/phase1-capture.sh body for MLX/OpenClaw diagnostics.
  • Reasoning: recommend an OpenClaw topology, routing, and governance stance for oc-mac-m5 from repo evidence.

Main Results

  • The 7B lane won overall, won throughput, and led on the benchmarked coding and reasoning tasks.
  • The larger models did not justify default promotion on this first pass.
  • The durable conclusion was that local-first on this node was already viable.

Run 2

Run 2 moved from “which model is best” toward “which model is best for which kind of work.” It expanded the problem set and added a broader role-separation question.

Models In Scope

Qwen2.5-7B-Instruct-4bit, Qwen2.5-32B-Instruct-4bit, Qwen2.5-Coder-32B-Instruct-4bit, and DeepSeek-R1-Distill-Qwen-32B-4bit.

Problems Run

  • Throughput: concise topology-difference bullets with warm-up excluded from scoring.
  • Coding: repair a broken benchmark harness with mislabeled timing, warm-up contamination, and malformed JSON handling defects.
  • Long-context reasoning: infer current versus planned oc-mac-m5 role using only local repo evidence.
  • Governance overclaim: distinguish safe claims from unsafe promotion-grade claims and identify proper document placement.

Main Results

  • The 7B lane stayed the best operational default.
  • The 32B instruct lane became the stronger hard-problem and reasoning option.
  • The coding lane was mixed rather than decisively specialist.
  • The benchmark program started to look more like routing by use case than a single leaderboard.

Run 3

Run 3 sharpened the local escalation threshold question. Frontier was unavailable, but the run still tested whether harder engineering and debugging tasks justified moving off the fast local default.

Models In Scope

Local models: Qwen2.5-7B-Instruct-4bit, Qwen2.5-Coder-32B-Instruct-4bit, and Qwen2.5-32B-Instruct-4bit. Frontier references were attempted but blocked.

Problems Run

  • Multi-file repair showdown: determine whether the coding-specialist lane materially beats the 7B default on bounded engineering repair.
  • Repo engineering helper: create or repair a benchmark integrity helper that checks artifacts, score CSVs, blocked task representation, and workspace boundaries.
  • Debugging contradictions: resolve inconsistent run state with malformed logs and overstated summaries.
  • Escalation decision: recommend routing thresholds by task class and identify what still needs frontier evidence.

Main Results

  • The 7B lane remained the best operational default and even scored extremely well across local tasks.
  • The 32B instruct lane became the clearer local reasoning/governance model.
  • The coder 32B lane stayed conditional rather than dominant.
  • The frontier comparison question remained open, which set up Run 4.
Run 3 summary page

Run 4

Run 4 is the most complete benchmark package so far. It directly tested the local-to-frontier escalation boundary using a structured prompt set, saved raw responses, normalized scored outputs, and chart-backed interpretation.

Models In Scope

Primary local lane: Qwen2.5-7B-Instruct-4bit, Qwen2.5-Coder-32B-Instruct-4bit, Qwen3.5-27B-4bit, and Qwen2.5-32B-Instruct-4bit. Frontier lane: gpt-5.2-codex and gpt-5.4.

Problems Run

  • Multi-file engineering repair: fix harness defects around timing labels, warm-up scoring, blocked summary omissions, and malformed JSON loss.
  • Benchmark integrity helper: build or repair a bounded helper that validates artifact completeness and score integrity.
  • Partial-failure debugging: identify contradictions, infer root-cause sequence, preserve evidence, and separate correcting results from rewriting history.
  • Escalation-boundary decision: recommend when to stay local, escalate to a stronger local specialist, or escalate to frontier.

Main Results

  • The 7B lane remained the best local operational default.
  • The coder 32B lane became the strongest local coding specialist.
  • The 32B instruct lane became the strongest local reasoning/governance lane.
  • Frontier produced stronger answers on some tasks, but not enough to justify universal escalation in this run.

What Exists Versus What Is Still Missing

Run 4 already has a curated prompt set, normalized data layer, raw outputs, scored artifacts, and charts. Runs 1 to 3 now have readable narrative summaries, but they do not yet have the same normalized prompt and scoring presentation layer. If this benchmark program continues, the next structural improvement should be a shared cross-run schema rather than more one-off prose.
