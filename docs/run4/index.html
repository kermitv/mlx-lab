Why Run 4 Matters

Earlier runs already suggested a local routing stack: use the fast 7B lane by default, escalate to the coder 32B lane for harder repair work, and prefer the 32B instruct lane for reasoning and governance-style tasks. Run 4 tested whether frontier changed that enough to justify routine escalation.

What Held Up

The local routing story remained coherent. The 7B lane still carried the operational default, and stronger local specialists still had clear role separation.

What Changed

Frontier did produce stronger answers on some tasks, especially under ambiguity, but the gain was not large enough in this run to displace the local-first routing recommendation.

What Stayed Bounded

These are mlx-lab-only findings. Memory looked satisfactory for the tested workloads, but process-level memory telemetry remained partial.

How To Read The Scores

The benchmark does not treat a single numeric score as a universal truth. The 1 to 10 scores are operator-oriented judgment scores built from saved raw responses, extracted answers, and per-task verdicts. They are useful for comparison inside this run, but they are not academic benchmarks and they should not be overread outside the tested task families.

Quality Score Meaning

The main quality axis reflects how well a model solved the actual task, not how polished the prose looked.

9 to 10: strong result with correct or near-correct handling, bounded claims, and useful operator value. 7 to 8: useful but with omissions, weaker precision, or some need for operator interpretation. 5 to 6: partially helpful, but too incomplete or error-prone to trust unattended. Below 5: materially weak, misleading, or not operationally dependable for the benchmarked task.

What “Quality” Covers Here

Run 4 quality was intentionally multidimensional, even when summarized into a smaller set of charts.

  • Root-cause identification and fix correctness for coding and debugging tasks.
  • Boundedness, governance fidelity, and overclaim avoidance for reasoning-heavy tasks.
  • Final-answer usefulness and maintainability, not just raw verbosity or confidence.

Operational Score Meaning

The operational axis is about unattended usefulness rather than answer quality alone.

  • Higher is better for latency, reliability, formatting consistency, and clean completion.
  • A model can score high on quality and still lose operationally if it is slower, retry-prone, or harder to trust unattended.
  • This is why the fastest local lane can remain the best default even when it is not the single strongest model on every task.

Reading Guide

Start with quality versus latency, then read the frontier-versus-local gap chart. Those two visuals carry most of the routing interpretation. The heatmap and quadrant view are there to test whether that reading stays coherent across all task families.

Heatmap Interpretation

Each cell represents the per-task quality result for one model on one benchmark family. Darker or stronger cells indicate higher quality within the Run 4 1 to 10 scale.

  • Read across a row to compare models on the same task family.
  • Read down a column to see whether a model is specialized or broadly consistent.
  • The important question is not “who is best overall,” but “where does a stronger model materially change the outcome?”

Quality vs Latency Interpretation

Points higher on the chart are better answers; points further left are faster. The useful region is the upper-left, where a model gives strong quality without paying unnecessary latency.

  • If a frontier point is only slightly higher but much slower, that usually does not justify routine escalation.
  • If a stronger local specialist moves meaningfully upward without extreme latency cost, that supports local-first escalation before frontier.

Gap and Quadrant Interpretation

The gap chart asks whether frontier materially beat the best local result by task class. The quadrant chart asks whether a model is strong as an unattended default or only strong when answer quality is the only priority.

  • A zero or near-zero frontier gap means local was already sufficient for that task class in this run.
  • A positive frontier gap matters most when it is paired with acceptable operational tradeoffs, not just a better raw answer.
